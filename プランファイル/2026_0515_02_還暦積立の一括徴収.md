# 2026_0515_02 還暦積立の一括徴収

## Context（背景・目的）

還暦積立は通常積立と異なり、社員が任意のタイミングでまとまった金額（例: 10,000 円）を支払うケースがある。
現状は徴収チェック画面でセッションごとに個別チェックする運用のため、複数月分をまとめて支払われた場合の操作が手間（複数セッションを開いて1件ずつチェック）になっている。

本対応では「還暦積立一括徴収」画面を新設し、社員と入金額を指定するだけで、**進行中の還暦積立セッション（IsCompleted=false）の古い月の未徴収から順に IsCollected を立てて消込する** 機能を提供する。

---

## 仕様確定事項（ユーザー確認済み）

| 項目 | 仕様 |
|---|---|
| 超過時の挙動 | 入力金額 > 未徴収合計 ならエラーで保存しない |
| 中途半端に余る場合 | ぴったり消化できなければエラーで保存しない（部分徴収不可） |
| 社員コンボ表示対象 | 進行中の還暦積立で未徴収レコードを持つ社員のみ |
| 上部メニュー位置 | 「徴収管理」と「収支帳簿」の間に独立リンクとして追加 |
| 領収書発行 | 行わない（要件外） |
| 収支帳簿連動 | 行わない（既存どおりセッション締切時に Complete で自動入金） |
| IsExcluded | 還暦積立では使用しないため対象外（除外フラグONのレコードは消込対象外） |

### 2. Controller アクション追加

`LPG_Tsumitate_Kanri2/Controllers/CollectionsController.cs` の末尾付近（Reopen の隣あたり）に以下を追加。

```csharp
// GET: /Collections/BulkRetirement
public async Task<IActionResult> BulkRetirement()
{
    var vm = new BulkRetirementViewModel
    {
        Candidates = await BuildCandidatesAsync()
    };
    return View(vm);
}

// POST: /Collections/BulkRetirement
[HttpPost, ValidateAntiForgeryToken]
public async Task<IActionResult> BulkRetirement(BulkRetirementViewModel input)
{
    input.Candidates = await BuildCandidatesAsync();

    if (!ModelState.IsValid)
        return View(input);

    // 対象: 指定社員 × 還暦積立(2) × 進行中(!IsCompleted) × 未徴収(!IsCollected) × 除外なし
    var records = await _db.CollectionRecords
        .Include(r => r.Session)
        .Where(r => r.EmployeeId == input.EmployeeId
            && r.Session.SavingsTypeId == 2
            && !r.Session.IsCompleted
            && !r.IsCollected
            && !r.IsExcluded)
        .OrderBy(r => r.Session.Year)
        .ThenBy(r => r.Session.Month)
        .ThenBy(r => r.RecordId)
        .ToListAsync();

    if (records.Count == 0)
    {
        ModelState.AddModelError(nameof(input.EmployeeId), "選択された社員には進行中の未徴収レコードがありません。");
        return View(input);
    }

    var pendingTotal = records.Sum(r => r.ExpectedAmount);
    if (input.Amount > pendingTotal)
    {
        ModelState.AddModelError(nameof(input.Amount),
            $"未徴収合計（{pendingTotal:N0} 円）を超えています。");
        return View(input);
    }

    // 古い順にぴったり消化できるか判定（部分徴収不可）
    var consumed = new List<CollectionRecord>();
    int remaining = input.Amount!.Value;
    foreach (var r in records)
    {
        if (remaining == 0) break;
        if (r.ExpectedAmount > remaining)
            break; // ぴったり消化できないので終了
        consumed.Add(r);
        remaining -= r.ExpectedAmount;
    }

    if (remaining != 0)
    {
        ModelState.AddModelError(nameof(input.Amount),
            "入力金額では古い未徴収から順にぴったり消し込めません。月額の組み合わせに合う金額を入力してください。");
        return View(input);
    }

    var now = DateTime.Now;
    foreach (var r in consumed)
    {
        r.IsCollected = true;
        r.CollectedAt = now;
    }
    await _db.SaveChangesAsync();

    TempData["BulkRetirementResult"] =
        $"{consumed.Count} 件 / 合計 {input.Amount:N0} 円を消し込みました。";
    return RedirectToAction(nameof(BulkRetirement));
}

private async Task<List<RetirementCandidate>> BuildCandidatesAsync()
{
    // 進行中の還暦積立で未徴収レコードを持つ社員のみ
    var rows = await _db.CollectionRecords
        .Where(r => r.Session.SavingsTypeId == 2
            && !r.Session.IsCompleted
            && !r.IsCollected
            && !r.IsExcluded
            && r.Employee.IsActive
            && !r.Employee.IsOnLeave)
        .GroupBy(r => new { r.EmployeeId, r.Employee.EmployeeNo, r.Employee.FullName })
        .Select(g => new RetirementCandidate
        {
            EmployeeId = g.Key.EmployeeId,
            EmployeeNo = g.Key.EmployeeNo,
            FullName = g.Key.FullName,
            PendingTotal = g.Sum(r => r.ExpectedAmount)
        })
        .OrderBy(c => c.EmployeeNo)
        .ToListAsync();
    return rows;
}
```

**消込ロジックの考え方**:

- 進行中 × 還暦積立 × 未徴収 × 除外なしのレコードを `Year, Month, RecordId` 昇順で取得
- 入力金額 > 未徴収合計 → エラー
- 古い順に1件ずつ `ExpectedAmount` を残額から引いていき、残額が 0 でぴったり止まらない場合（途中で「次の ExpectedAmount > 残額」が起きる、または最後まで回しても残額 > 0）→ エラー
- 0 で止まれば、消化した全レコードに `IsCollected=true` / `CollectedAt=DateTime.Now` を設定して保存

**ぴったり消化の例**:

- 月額 500 円のレコードが 4 件あり、計 2,000 円 → 入力 1,500 円 OK（3件消化、残 0）
- 月額 500 円 × 2 件、月額 2,000 円 × 1 件で計 3,000 円 → 入力 1,000 円 OK（500+500=1,000）／入力 800 円 NG（1件目500、残300<次の500）／入力 1,300 円 NG（500+500=1,000、残300<次の2,000）

## 既存仕様に与える影響

- セッション締切（Complete アクション）のロジックは未変更。一括徴収で `IsCollected=true` になったレコードは、後でそのセッションが Complete された際に、既存どおり徴収済合計として LedgerEntry に反映される。
- 徴収チェック画面（`/Collections/Details/{id}`）には影響なし。一括徴収後にそのセッションの画面を開くと、消込済の社員が「徴収済み」チェックされた状態で表示される。
- 通常積立（SavingsTypeId=1）には影響なし。

## 検証手順

ローカルビルド・実行は作業者（ユーザー）が行う。Claude 側ではビルド・起動は行わない。

1. `dotnet build` でビルドエラーなしを確認
2. `dotnet run --project LPG_Tsumitate_Kanri2/LPG_Tsumitate_Kanri2.csproj` で起動
3. 上部メニューに「還暦積立一括徴収」が表示され、`/Collections/BulkRetirement` に遷移できること
4. 還暦積立の進行中セッションを2つ以上作り、ある社員に未徴収レコードを複数月分残した状態を用意
5. 画面でその社員をコンボから選択し、未徴収の月額合計ぴったりの金額（例: 月500円×3件なら1,500円）を入力 → 保存
   - 成功メッセージ「3 件 / 合計 1,500 円を消し込みました。」が表示されること
   - 該当セッションの Details 画面でその社員が「徴収済み」チェック済みになっていること
6. **超過エラー検証**: 未徴収合計を超える金額を入力 → エラーメッセージ「未徴収合計（◯◯ 円）を超えています。」が出て保存されないこと
7. **ぴったり不一致エラー検証**: 月額500円×2件のみ未徴収の社員に800円入力 → エラー「古い未徴収から順にぴったり消し込めません…」が出て保存されないこと
8. **対象者フィルタ検証**: 未徴収のない社員はコンボに表示されないこと。未徴収を持つ社員のみが「（未徴収 ◯◯ 円）」付きで並ぶこと
9. **古い順の検証**: 2025/12月分と2026/3月分の未徴収がある社員に 12月分相当の金額を入れたら 12月分のみが消し込まれること
10. **進行中フィルタ検証**: 締切済みセッションの未徴収は対象外。締切済みのレコードが影響を受けないこと
11. スマホ表示でフォームが縦に並び、PC 表示で崩れないこと

## 補足

- ぴったり消化のロジックは「古い順 + greedy」で十分。CollectionRecord の ExpectedAmount は社員ごとに固定（ContributionAmountRule から計算済）で、進行中セッション内で全件同じ金額になることが多いため、`残額 % ExpectedAmount == 0` で済むケースが大半。ただし社員の属性変更等で月によって金額が変わる可能性に備え、ループでぴったり判定する設計とする。
- 同一社員の同一月に複数 CollectionRecord は発生し得ない（`UNIQUE(SessionId, EmployeeId)` 相当の運用）が、ソートキーに `RecordId` を含めて安定性を担保。
- データモデル変更がないためマイグレーションは不要。
